# Mint Architecture V1 — Trade Artifact Model

**Status:** FROZEN  
**Date:** 2026-03-16  

---

## Design Principles

1. **Minimal on-chain footprint** — store only what's needed for verification, keep everything else off-chain
2. **Standard NFT** — artifact must show up in wallets, explorers, and marketplaces as a recognizable token
3. **Claim-gated minting** — no mint without a valid Ed25519 signature from the trading wallet
4. **Permanent off-chain storage** — receipt data and rendered card on Arweave (immutable, cheap)
5. **Agent-first CLI** — every step scriptable, no browser wallets required

---

## 1. Artifact Structure

### What Gets Minted

A single **non-fungible token** (1 per closed trade cycle) minted to `claim_recipient`.

### On-Chain Data (PDA — Receipt Anchor)

The Anchor program creates a PDA for each minted receipt:

```
Seeds: ["receipt", verification_hash]
```

| Field | Type | Size | Description |
|---|---|---|---|
| `verification_hash` | `[u8; 32]` | 32 | SHA-256 of receipt canonical fields |
| `metadata_hash` | `[u8; 32]` | 32 | SHA-256 of the Arweave metadata JSON content. Binds the on-chain anchor to the exact off-chain metadata, surviving hosting changes. |
| `trader_wallet` | `Pubkey` | 32 | Original trading wallet |
| `claim_recipient` | `Pubkey` | 32 | Authorized mint destination |
| `claim_signature` | `[u8; 64]` | 64 | Ed25519 signature of claim message |
| `status` | `u8` | 1 | `0=verified`, `1=verified_mixed_quote` |
| `program_version` | `u8` | 1 | Protocol version that minted this receipt. V1 = `1`. Allows future verifiers to understand which rules applied at mint time. |
| `mint` | `Pubkey` | 32 | NFT mint address |
| `minted_at` | `i64` | 8 | Unix timestamp of mint tx |
| `bump` | `u8` | 1 | PDA bump seed |

**Total PDA size: ~235 bytes + 8 byte discriminator = ~243 bytes**

Rent cost: ~0.0025 SOL (one-time, rent-exempt)

### `metadata_hash` Derivation

The `metadata_hash` is the SHA-256 of the **exact bytes** of the Arweave metadata JSON file (the NFT metadata JSON, not the full receipt JSON). This is computed before upload:

```javascript
const metadataJson = JSON.stringify(metadataObject);
const metadataHash = sha256(Buffer.from(metadataJson));
```

This ensures:
- The on-chain anchor can always be matched to the correct off-chain metadata
- If Arweave hosting changes or the URI format evolves, the hash still validates
- A verifier can download the metadata from any source and confirm it matches the on-chain hash

### `program_version` Values

| Value | Meaning |
|---|---|
| `1` | V1 protocol rules (current). WACB accounting, Helius ingest, Token-2022 NFT, soul-bound. |
| `2+` | Reserved for future protocol versions with different rules. |

When verifying a receipt, the verifier should check `program_version` and apply the corresponding validation rules. This prevents a V2 verifier from incorrectly rejecting a valid V1 receipt due to rule changes.

### On-Chain NFT Metadata

Using **Token-2022 with Metadata Extension** (no Metaplex dependency):

| Metadata Field | Value |
|---|---|
| `name` | `Trade Receipt #{receipt_id short}` |
| `symbol` | `TREC` |
| `uri` | Arweave URI to off-chain JSON metadata |

The NFT is **non-transferable in V1** (soul-bound to `claim_recipient`). This prevents receipt trading/speculation. Implemented via Token-2022's `NonTransferable` extension.

### Off-Chain Data (Arweave)

Two files uploaded to Arweave per receipt:

**1. Metadata JSON** (follows Metaplex JSON standard for wallet compatibility):


## 2. On-Chain Program Design

### Program: `trade_artifact`

A single Anchor program with 2 instructions:

#### Instruction 1: `mint_receipt`

The core instruction. Verifies claim, creates PDA, mints NFT.

```rust
pub fn mint_receipt(
    ctx: Context<MintReceipt>,
    verification_hash: [u8; 32],
    metadata_hash: [u8; 32],       // SHA-256 of Arweave metadata JSON
    claim_signature: [u8; 64],
    claim_message: Vec<u8>,        // canonical claim message bytes
    status: u8,                     // 0=verified, 1=verified_mixed_quote
    metadata_uri: String,           // Arweave URI
    receipt_name: String,           // "Trade Receipt #0004"
) -> Result<()>
```

**Verification steps performed on-chain:**

1. **Signature check** — Verify `claim_signature` over `claim_message` using `trader_wallet` public key via Ed25519 program (Solana native `ed25519_program`)
2. **Message integrity** — Parse `claim_message` to extract and validate:
   - `verification_hash` matches instruction arg
   - `wallet` matches `trader_wallet` account
   - `chain` equals `"solana"`
   - `claim_recipient` matches the `recipient` account
3. **Uniqueness** — PDA derivation from `["receipt", verification_hash]` ensures one mint per receipt (duplicate mint attempts fail because PDA already exists)
4. **Mint NFT** — Create Token-2022 mint + metadata + NonTransferable, mint 1 token to `claim_recipient`'s ATA
5. **Store anchor** — Write PDA with receipt anchor data including `metadata_hash` and `program_version = 1`

#### Instruction 2: `verify_receipt`

Read-only verification. No state changes. Returns the PDA data for a given `verification_hash`.

```rust
pub fn verify_receipt(
    ctx: Context<VerifyReceipt>,
    verification_hash: [u8; 32],
) -> Result<ReceiptAnchor>
```

This allows anyone to check if a receipt has been minted on-chain and retrieve its anchor data, including the `metadata_hash` for off-chain content verification and `program_version` for rule-set identification.

#### No Close/Delete Instruction

Trade receipts are **immutable historical artifacts**. There is no instruction to close, delete, or modify a receipt PDA. Once minted, the receipt exists permanently on-chain.

If a receipt contains an error, the correct approach is to generate a new receipt with corrected data, producing a new `verification_hash` and a new PDA. The original receipt remains as historical record. Future versions may add a `superseded_by` field to link corrected receipts.

### Account Layout

```
MintReceipt {
    #[account(mut)] payer: Signer,              // pays rent + tx fees
    trader_wallet: UncheckedAccount,             // trading wallet (verified via sig)
    #[account(mut)] recipient: SystemAccount,    // claim_recipient
    #[account(init, pda)] receipt_anchor: Account<ReceiptAnchor>,
    #[account(init)] nft_mint: Account<Mint>,    // Token-2022
    #[account(init)] recipient_ata: Account<TokenAccount>,
    token_program: Program<Token2022>,
    system_program: Program<System>,
    // ... associated token, rent, etc.
}
```

### Program Size Estimate

2 instructions, Ed25519 verification, Token-2022 CPI calls. Estimated deployed program size: **~12-18 KB**. Well within Solana's limits.

### Cost Per Mint

| Component | Cost (SOL) |
|---|---|
| PDA rent (243 bytes, rent-exempt) | ~0.0025 |
| NFT mint rent (Token-2022) | ~0.002 |
| ATA rent | ~0.002 |
| Transaction fees | ~0.000015 |
| **Total** | **~0.0065 SOL** |

## 3. Metadata Layout

### Storage: Arweave

Arweave is chosen for V1 because:
- **Permanent** — data is stored forever with a one-time payment
- **Cheap** — ~$0.001-0.01 per file at current rates
- **Standard** — Solana NFT ecosystem already uses Arweave URIs
- **Deterministic** — content-addressed (same data = same txid)

### Upload Flow

```
1. Render receipt card → PNG file
2. Upload PNG to Arweave → ar://<png_txid>
3. Build metadata JSON (includes ar:// PNG reference)
4. Upload full receipt JSON to Arweave → ar://<receipt_txid>  
5. Upload metadata JSON to Arweave → ar://<metadata_txid>
6. Use ar://<metadata_txid> as NFT metadata URI
```

### Alternative: Deterministic Reconstruction (V2)

In future versions, the metadata URI could point to a reconstruction endpoint instead of static storage. Given the receipt JSON and the engine code, the PNG can be deterministically regenerated. This would eliminate storage costs entirely, but requires a running service.

V1 uses static Arweave storage for simplicity and permanence.

### Content Addressing

Since the receipt data is deterministic, the same receipt will always produce the same Arweave content. This means:
- Re-uploading is idempotent
- Multiple parties uploading the same receipt converge on the same content
- The verification hash in the receipt can be cross-referenced with the on-chain PDA

### Step-by-Step

| Step | Actor | Action | Output |
|---|---|---|---|
| 1 | Engine | Run pipeline on wallet | `receipts.jsonl` + PNG cards |
| 2 | Engine | Upload PNG + JSON + metadata to Arweave | Arweave transaction IDs |
| 3 | Trader | Sign claim message with wallet | Ed25519 signature |
| 4 | Payer | Submit `mint_receipt` transaction | Receipt PDA + NFT |
| 5 | Anyone | Call `verify_receipt` or read PDA | Verification result |

**Note:** The payer (step 4) can be anyone — the trader, an agent, or a relayer. The claim signature proves authorization regardless of who submits the transaction.

## 5. CLI Interface

### `tae` — Trade Artifact Engine CLI

All commands operate on the workspace's `data/` directory and Solana network.

#### Receipt Generation

```bash
# Run full pipeline for a wallet
tae generate --wallet <pubkey> [--max-txns 10000]

# Output: data/receipts/receipts.jsonl + data/renders/*.png
```

#### Claim Production

```bash
# Produce claim message for signing
tae claim prepare --receipt-id receipt_0004_DitHyRMQ \
                   --recipient <pubkey>

# Output: canonical claim message (UTF-8) to stdout or file
# The agent/user signs this with their wallet

# Assemble claim object from message + signature
tae claim assemble --receipt-id receipt_0004_DitHyRMQ \
                    --recipient <pubkey> \
                    --signature <base58_sig>

# Output: data/claims/claim_0004_DitHyRMQ.json
```

#### Verification

```bash
# Verify receipt JSON (hash re-derivation)
tae verify receipt --receipt-id receipt_0004_DitHyRMQ

# Verify claim signature
tae verify claim --claim-id claim_0004_DitHyRMQ

# Verify on-chain anchor (checks PDA exists + matches)
tae verify on-chain --verification-hash <hex>

# Full verification (re-run engine from chain + compare)
tae verify full --receipt-id receipt_0004_DitHyRMQ
```

#### Upload

```bash
# Upload receipt assets to Arweave
tae upload --receipt-id receipt_0004_DitHyRMQ \
           [--arweave-key <path>]

# Output: Arweave txids for PNG, JSON, metadata
```

#### Mint

```bash
# Submit mint transaction
tae mint --receipt-id receipt_0004_DitHyRMQ \
         --claim claim_0004_DitHyRMQ.json \
         --payer <keypair_path> \
         [--network devnet|mainnet]

# Output: transaction signature, PDA address, NFT mint address
```

#### Inspect

```bash
# Show receipt details with sanity checks
tae inspect --receipt-id receipt_0004_DitHyRMQ

# List all receipts for a wallet
tae list --wallet <pubkey>

# Show on-chain anchor for a receipt
tae inspect on-chain --verification-hash <hex>
```

### Agent Integration

The CLI is designed for scriptable automation. An agent workflow:

```javascript
// 1. Generate receipts
await exec('tae generate --wallet <pubkey> --max-txns 10000');

// 2. For each receipt, prepare claim
const msg = await exec('tae claim prepare --receipt-id ... --recipient ...');

// 3. Sign claim (agent has wallet access)
const sig = await signMessage(msg, walletKeypair);

// 4. Assemble claim
await exec(`tae claim assemble --receipt-id ... --recipient ... --signature ${sig}`);

// 5. Upload to Arweave
await exec('tae upload --receipt-id ...');

// 6. Mint on-chain
await exec('tae mint --receipt-id ... --claim ... --payer ...');
```

All steps are independent and resumable. If upload fails, re-run upload. If mint fails, re-run mint. The verification hash ensures idempotency.

## Open Questions for V1

| Question | Answer | Notes |
|---|---|---|
| NFT transferability | Soul-bound (NonTransferable) | Prevents receipt trading. Can revisit in V2. |
| Who pays mint cost? | Payer can be anyone | Trader, agent, or relayer |
| Arweave funding | Bundlr/Irys for SOL-denominated uploads | No AR token needed |
| Multiple mints of same receipt? | Blocked by PDA uniqueness | One artifact per verification_hash |
| Receipt updates/corrections? | New receipt + new hash | Old PDA remains as immutable record; `superseded_by` in V2 |
| Receipt deletion? | Not supported | Immutable historical artifacts. No close instruction. |
| Token-2022 vs Metaplex Core? | Token-2022 | No external program dependency, native Solana |
| Metadata durability? | `metadata_hash` in PDA | Content hash survives hosting changes |
| Protocol versioning? | `program_version` in PDA | Future verifiers know which rules applied |

## What V1 Does NOT Include

- Transferable receipts / trading
- Receipt collections or portfolios
- On-chain PnL aggregation
- Reputation scoring
- Multi-chain support
- USD normalization for MIXED quotes
- Delegation-based claiming (V2)
- On-chain receipt data beyond the anchor (full data stays off-chain)

## Summary

The V1 mint architecture is deliberately minimal:

- **~243 bytes on-chain** per receipt (PDA anchor with `metadata_hash` + `program_version`)
- **~0.0065 SOL** per mint
- **3 Arweave files** per receipt (PNG, JSON, metadata)
- **1 Anchor program** with 2 instructions (`mint_receipt`, `verify_receipt`)
- **Immutable** — no delete/close instruction; receipts are permanent historical artifacts
- **Soul-bound NFT** visible in wallets (Token-2022 NonTransferable)
- **Content-verified** — `metadata_hash` binds on-chain anchor to exact off-chain data
- **Version-tagged** — `program_version` enables forward-compatible verification
- **Full CLI** for agents and scripts
- **Claim-gated** — no mint without trader wallet signature
