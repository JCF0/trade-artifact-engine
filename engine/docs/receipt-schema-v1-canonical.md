# TradeReceipt Schema V1 — Canonical Specification

**Status:** FROZEN  
**Version:** 1.0  
**Date:** 2026-03-16  

---

## Overview

A TradeReceipt is a structured JSON object representing a single closed spot trade cycle on Solana. It captures the full lifecycle from entry to exit with deterministic, reproducible calculations derived entirely from on-chain data.

Receipts are generated by the Trade Artifact Engine and are intended to be:
- **Verifiable** — any party can re-derive the verification hash from on-chain data
- **Deterministic** — same inputs always produce identical output
- **Self-contained** — receipt includes all data needed for independent verification

---

## Required Fields

### Identity

| Field | Type | Description |
|---|---|---|
| `receipt_id` | string | Unique receipt identifier. Format: `receipt_{NNNN}_{mint8}` |
| `receipt_version` | string | Schema version. Must be `"1.0"` for V1 receipts |
| `cycle_id` | string | Reference to the internal trade cycle that produced this receipt |

### Context

| Field | Type | Description |
|---|---|---|
| `wallet` | string | Solana wallet public key (base58) |
| `chain` | string | Always `"solana"` in V1 |
| `token_mint` | string | SPL token mint address (base58) |
| `status` | string | Receipt status (see Status Values below). **Included in verification hash.** |
| `accounting_method` | string | Always `"weighted_average_cost_basis"` in V1 |

### Pricing

| Field | Type | Description |
|---|---|---|
| `avg_entry_price` | number | Weighted average entry price in quote currency per token |
| `avg_exit_price` | number | Weighted average exit price in quote currency per token |
| `quote_currency` | string | Quote currency mint address, OR `"MIXED"` (see Quote Currency Handling) |

### PnL

| Field | Type | Description |
|---|---|---|
| `total_cost_basis` | number | Total quote currency spent on entry: `Σ(entry_tx.quote_amount)` |
| `total_exit_proceeds` | number | Total quote currency received on exit: `Σ(exit_tx.quote_amount)` |
| `realized_pnl` | number | `total_exit_proceeds - total_cost_basis` |
| `realized_pnl_pct` | number | `(realized_pnl / total_cost_basis) × 100` |

### Position

| Field | Type | Description |
|---|---|---|
| `total_bought` | number | Total tokens acquired across all entry transactions |
| `total_sold` | number | Total tokens disposed across all exit transactions |
| `peak_position` | number | Maximum cumulative token holding at any point during cycle |
| `remaining_balance` | number | `total_bought - total_sold` (should be near zero for closed cycles) |
| `num_buys` | number | Count of entry (buy) transactions |
| `num_sells` | number | Count of exit (sell) transactions |

### Timing

| Field | Type | Description |
|---|---|---|
| `opened_at` | number | Unix timestamp (seconds) of first entry transaction |
| `closed_at` | number | Unix timestamp (seconds) of last exit transaction that closed the cycle |
| `hold_time_seconds` | number | `closed_at - opened_at` |

### Transaction References

| Field | Type | Description |
|---|---|---|
| `entry_txs` | array | Entry transactions (see Transaction Object below) |
| `exit_txs` | array | Exit transactions (see Transaction Object below) |

### Metadata

| Field | Type | Description |
|---|---|---|
| `generated_at` | number | Unix timestamp (seconds) when receipt was generated. **Not included in verification hash.** |
| `verification_hash` | string | SHA-256 hex digest over canonical fields (see Verification Hash Spec) |

## Transaction Object

Each element in `entry_txs` and `exit_txs`:

| Field | Type | Description |
|---|---|---|
| `tx_hash` | string | Solana transaction signature (base58) |
| `timestamp` | number | Unix timestamp (seconds) of the transaction |
| `amount` | number | Token amount (decimal-normalized) |
| `quote_amount` | number | Quote currency amount (decimal-normalized) |

## Optional Fields (V1)

These fields MAY be present but are not required for verification:

| Field | Type | Description |
|---|---|---|
| `token_symbol` | string | Human-readable token symbol (best effort, not authoritative) |
| `quote_symbol` | string | Human-readable quote symbol (e.g. "SOL", "USDC") |
| `tags` | array[string] | User or engine-applied tags (e.g. `["pump.fun", "meme"]`) |
| `notes` | string | Free-text annotation |

Optional fields are NEVER included in the verification hash.

## Status Values

| Status | Meaning |
|---|---|
| `verified` | Trade cycle fully reconstructed with high confidence. Single quote currency, all swaps matched, balance closed cleanly. |
| `verified_mixed_quote` | Trade cycle fully reconstructed but entry/exit transactions used different quote currencies. PnL is computed by summing raw quote amounts across currencies. Directionally correct but not unit-precise. |
| `ambiguous` | Trade cycle detected but some data is uncertain (e.g. missing swap decoding, partial parse). Receipt generated but flagged. |
| `unsupported` | Trade type not handled in V1. No receipt generated. |

### Status Assignment Rules

1. If all `entry_txs` and `exit_txs` used the same `quote_mint` → `verified`
2. If transactions used different `quote_mint` values → `verified_mixed_quote`
3. If any swap event relied on fallback extraction AND the result is ambiguous → `ambiguous`
4. Receipts are only generated for `verified` and `verified_mixed_quote` cycles

### Status in Verification Hash

The `status` field is **included in the verification hash** because `verified` and `verified_mixed_quote` represent materially different interpretations of the same trade data. A receipt's proof must bind to the classification the engine assigned. If the engine's classification logic changes in a future version, the receipt must be re-generated with a new hash — it cannot be silently reclassified.

## Quote Currency Handling

### Single Quote (preferred)

When all entry and exit transactions in a cycle use the same quote currency (e.g. all SOL or all USDC), `quote_currency` is set to that mint address. Accounting is exact.

### Mixed Quote

When transactions in a cycle use different quote currencies (e.g. some buys in SOL, some in USDC):

- `quote_currency` is set to `"MIXED"`
- `status` is set to `"verified_mixed_quote"` (not `"verified"`)
- `total_cost_basis` and `total_exit_proceeds` are computed by summing raw quote amounts as-is
- `avg_entry_price` and `avg_exit_price` are derived from these sums
- The resulting PnL is **directionally correct** but mixes units

**V1 does not normalize mixed quotes to a common denomination.** This is a known limitation. Future versions may normalize to USD using historical price data at time of each swap.

### Canonical Quote Mints (V1)

| Symbol | Mint |
|---|---|
| SOL | `So11111111111111111111111111111111111111112` |
| USDC | `EPjFWdd5AufqSSqeM2qN1xzybapC8G4wEGGkZwyTDt1v` |
| USDT | `Es9vMFrzaCERmJfrF4H2FYD4KCoNkY11McCe8BenwNYB` |

Any other mint appearing as the quote side of a swap is treated as a non-standard quote and will trigger `MIXED` if combined with a canonical quote.

## Precision Rules

### Two-Layer Model: Raw (Hash) vs Display (Rendering)

The engine computes all accounting values as IEEE 754 double-precision floats. These values exist at two layers:

**Raw values (used in verification hash):**
- The engine's internal computation results with full double-precision
- These are the canonical source of truth
- The hash is computed from these values BEFORE any display rounding

**Display values (stored in receipt JSON):**
- Derived from raw values by applying display formatting rules
- Used for JSON serialization and rendering
- MAY differ from raw values by rounding

### Display Formatting Rules

| Field | Formatting | Example |
|---|---|---|
| `avg_entry_price` | `toPrecision(12)` | `0.0392993741894` |
| `avg_exit_price` | `toPrecision(12)` | `0.0561599974612` |
| `realized_pnl_pct` | `toPrecision(6)` | `42.903` |
| `total_cost_basis` | `toPrecision(12)` | `1000.954031` |
| `total_exit_proceeds` | `toPrecision(12)` | `1430.393664` |
| `realized_pnl` | `toPrecision(12)` | `429.439633` |
| `total_bought` | `toFixed(10)` | `25469.9738010000` |
| `total_sold` | `toFixed(10)` | `25469.9738010000` |
| `peak_position` | `toFixed(10)` | `25469.9738010000` |
| `remaining_balance` | `toFixed(10)` | `0.0000000000` |

### Rendering Display Rules (for PNG/UI)

| Field | Render Rule |
|---|---|
| Prices | If < 0.0001: scientific notation (4 sig figs). If < 1: 6 decimals. Otherwise: 4 decimals. |
| `realized_pnl` | 6 decimal places with sign prefix (`+` / `-`) |
| `realized_pnl_pct` | 3 decimal places with sign prefix and `%` suffix |
| Cost basis / proceeds | 6 decimal places |
| Token amounts | Up to 10 decimal places, trailing zeros trimmed |
| Timestamps | ISO 8601 UTC for display, Unix seconds in JSON |
| `hold_time_seconds` | Contextual: seconds / minutes / hours / days |

## Verification Hash Specification

### Algorithm

SHA-256

### Canonical Raw Values

The verification hash is computed over the engine's **raw accounting values** — the full double-precision results of the PnL computation BEFORE any display formatting is applied.

This design means:
- Display formatting changes (e.g. changing `toPrecision(12)` to `toPrecision(14)`) will **never** break existing hashes
- A verifier re-running the engine from on-chain data will produce the same raw values and thus the same hash
- The hash is bound to the mathematical result, not its textual representation

### Raw Value Computation

```javascript
// These are computed by the PnL engine as standard IEEE 754 doubles
const rawTotalCostBasis    = entryTxs.reduce((s, t) => s + t.quote_amount, 0);
const rawTotalExitProceeds = exitTxs.reduce((s, t) => s + t.quote_amount, 0);
const rawEntryPriceAvg     = rawTotalCostBasis / totalBought;
const rawExitPriceAvg      = rawTotalExitProceeds / totalSold;
```

These raw values are used directly in the hash payload — no `toPrecision()`, no `toFixed()`, no rounding.

### Input Construction

The hash is computed over a JSON-serialized array of canonical fields in this exact order:

```javascript
const payload = JSON.stringify([
  receipt.wallet,              // string: wallet public key
  receipt.chain,               // string: "solana"
  receipt.token_mint,          // string: token mint address
  entryTxHashes,               // string[]: entry tx signatures, SORTED ascending
  exitTxHashes,                // string[]: exit tx signatures, SORTED ascending
  rawEntryPriceAvg,            // number: RAW double-precision, no display rounding
  rawExitPriceAvg,             // number: RAW double-precision, no display rounding
  receipt.accounting_method,   // string: "weighted_average_cost_basis"
  receipt.receipt_version,     // string: "1.0"
  receipt.status,              // string: "verified" or "verified_mixed_quote"
]);

const hash = sha256(payload).hex();
```

### Hash Rules

1. Transaction hashes are **sorted lexicographically ascending** before hashing. This ensures order-independence.
2. Price values are the **raw double-precision results** from the PnL engine, not the display-formatted values stored in the receipt JSON. When `JSON.stringify` serializes a JavaScript `number`, it uses the shortest representation that uniquely identifies the IEEE 754 value — this is deterministic across conformant engines.
3. `status` is **included** because `verified` vs `verified_mixed_quote` represents a materially different interpretation of the trade. The proof must bind to the classification.
4. `generated_at` is **excluded** — it is metadata about when the receipt was created, not about the trade.
5. Optional fields are **excluded** — they do not affect verification.

### Receipt Storage of Hash Inputs

To enable verification without re-running the engine, the receipt stores the raw price values in a dedicated field:

| Field | Type | Description |
|---|---|---|
| `_hash_inputs` | object | Raw values used in hash computation (see below) |

```json
{
  "_hash_inputs": {
    "raw_entry_price_avg": 0.039299374189350795,
    "raw_exit_price_avg": 0.05615999746118498
  }
}
```

These values are the engine's unrounded doubles. They exist solely to allow hash re-derivation from the receipt without re-running the full pipeline. They are NOT used for display.

A full verifier who re-runs the engine from on-chain data should arrive at the same raw values. If they don't match, the receipt may have been tampered with.

### Verification Flows

**Quick verification (from receipt only):**

1. Read `_hash_inputs.raw_entry_price_avg` and `_hash_inputs.raw_exit_price_avg`
2. Extract and sort `entry_txs[].tx_hash` and `exit_txs[].tx_hash`
3. Construct the JSON array using raw values + receipt fields
4. Compute SHA-256
5. Compare to `verification_hash`

**Full verification (from on-chain data):**

1. Fetch all referenced transactions from Solana
2. Re-run normalization, cycle detection, and PnL calculation
3. Compare computed raw values against `_hash_inputs`
4. Compare computed display values against receipt fields
5. Re-derive hash from raw values and compare to `verification_hash`

## Cycle Close Threshold (Dust Rule)

A trade cycle is considered closed when:

```
remaining_balance < max(0.001 tokens, 0.1% × peak_position)
```

Where:
- `remaining_balance` = `|total_bought - total_sold|`
- `peak_position` = maximum cumulative holding at any point

This threshold is stored in the receipt via `peak_position` and `remaining_balance`, making it independently verifiable.

---

## Immutability

Once a receipt is generated and its verification hash is computed, no field included in the hash may be modified. If a correction is needed, a new receipt must be generated with a new `receipt_id` and the old receipt should be marked as superseded (mechanism TBD in future versions).
